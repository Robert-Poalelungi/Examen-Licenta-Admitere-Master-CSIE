## Întrebarea 1

**Cerința:** Fie tabela `ANGAJATI(id_angajat NUMBER(5) PRIMARY KEY, nume VARCHAR2(30), prenume VARCHAR2(30), salariul NUMBER(10,2), data_angajare DATE, id_departament NUMBER(3))` cu 200 de rânduri. Se consideră secvența PL/SQL:

```sql
SET SERVEROUTPUT ON
BEGIN
  FOR rang IN (SELECT * FROM angajati
               WHERE id_departament=50
               AND salariul > (SELECT AVG(salariul) FROM angajati)) LOOP
    DBMS_OUTPUT.PUT_LINE('Angajatul '||rang.nume||' are salariul '||rang.salariul);
  END LOOP;
END;
/
```

**Variante:**
- a) se utilizează un cursor pentru afișarea angajaților din departamentul 50 care au salariile mai mari decât salariul mediu
- b) comanda este eronată deoarece nu este permisă utilizarea comenzii SELECT în FOR
- c) cursorul nu este declarat corespunzător și se va afișa INVALID_CURSOR
- d) lipsește DECLARE și blocul PL/SQL nu se va compila cu succes

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: a)**

**Explicație detaliată:**

Blocul PL/SQL folosește un **cursor FOR implicit** (cursor FOR loop cu SELECT inline). Aceasta este o caracteristică validă în PL/SQL: nu este nevoie de un DECLARE explicit pentru cursor, nici de OPEN/FETCH/CLOSE — Oracle gestionează totul automat.

- **Clauza FOR...IN (SELECT ...)** creează automat un cursor anonim, îl deschide, îl parcurge rând cu rând și îl închide la final.
- **Condiția WHERE** filtrează angajații din `id_departament = 50` cu salariul mai mare decât media generală a tuturor angajaților (subcerere corelată).
- DECLARE nu este obligatoriu dacă nu există variabile declarate — blocul poate începe direct cu BEGIN.
- Nu există erori de sintaxă sau de logică.

Variantele b), c), d) sunt greșite:
- b) este falsă — SELECT în FOR cursor loop este perfect valid.
- c) este falsă — cursorul implicit nu generează INVALID_CURSOR.
- d) este falsă — DECLARE este opțional în PL/SQL când nu sunt variabile locale.

</details>

## Întrebarea 2

**Cerința:** În C++, dacă o clasă are cel puțin o metodă virtual pură atunci ea devine:

**Variante:**
- a) derivată
- b) abstractă
- c) virtuală
- d) friend

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: b) abstractă**

**Explicație detaliată:**

O **metodă virtual pură** se declară cu sintaxa:
```cpp
virtual tip_returnat numeFunctie(parametri) = 0;
```

O clasă care conține **cel puțin o metodă virtual pură** devine **clasă abstractă**. Consecințele sunt:
1. **Nu se pot instanția direct** obiecte din clasa abstractă.
2. Clasele derivate **trebuie să implementeze** toate metodele virtuale pure, altfel și ele devin abstracte.
3. Clasa abstractă **poate fi folosită ca tip pointer/referință** pentru a lucra polimorfic cu obiectele derivate.

Celelalte variante sunt greșite:
- a) „derivată" — o clasă devine derivată când moștenește altă clasă, nu are legătură cu metodele virtuale pure.
- c) „virtuală" — nu există conceptul de „clasă virtuală" (există moștenire virtuală, alt concept).
- d) „friend" — `friend` este o relație de acces, nu legată de metode virtuale pure.

</details>

## Întrebarea 3

**Cerința:** Se consideră secvența PL/SQL:

```sql
SET SERVEROUTPUT ON
DECLARE
  punctaj_examen NUMBER;
BEGIN
  IF punctaj_examen < 60 THEN
    dbms_output.put_line('Respins');
  ELSE
    dbms_output.put_line('Admis');
  END IF;
END;
/
```

**Variante:**
- a) blocul PL/SQL generează eroare din cauza utilizării incorecte a structurii alternative
- b) se afișează 'Respins'
- c) se afișează 'Admis'
- d) blocul PL/SQL rulează cu succes, dar nu se va afișa nimic

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: c) se afișează 'Admis'**

**Explicație detaliată:**

Variabila `punctaj_examen` este declarată ca `NUMBER` **fără valoare de inițializare** → valoarea sa este **NULL**.

În PL/SQL, evaluarea `NULL < 60` returnează **NULL** (nu TRUE, nu FALSE). Comportamentul structurii IF-THEN-ELSE când condiția este NULL:

- **Condiția nu este TRUE** → ramura THEN nu se execută.
- **Oracle PL/SQL tratează NULL ca non-TRUE** → controlul trece la ramura **ELSE**.
- Prin urmare, se execută `dbms_output.put_line('Admis')`.

Cheia de înțeles: în PL/SQL, `IF <conditie_NULL> THEN ... ELSE ... END IF;` — ramura ELSE se execută, deoarece condiția nu a fost TRUE. Aceasta diferă de SQL pur unde NULL se propagă ca UNKNOWN.

Variantele greșite:
- a) Nu există eroare de sintaxă — IF-THEN-ELSE este corect utilizat.
- b) 'Respins' s-ar afișa dacă `NULL < 60` ar fi TRUE, ceea ce nu este cazul.
- d) Ar fi corect dacă nu ar exista ramura ELSE; dar ELSE există și preia controlul.

</details>

**Cerința:** Fie tabelele `PRODUSE(id_produs, denumire_produs, descriere, categorie, pret_lista)` și `RAND_COMENZI(id_produs, id_comanda, pret, cantitate)`. Se consideră comanda:

```sql
UPDATE produse p
SET pret_lista = pret_lista * 0.9
WHERE pret_lista > (SELECT AVG(pret) FROM rand_comenzi WHERE id_produs = p.id_produs)
  AND categorie IN (SELECT categorie FROM produse
                    WHERE LOWER(denumire_produs) LIKE '%router%');
```

**Variante:**
- a) se actualizează prețul produselor care au `pret_lista` mai mare decât prețul mediu de vânzare din `RAND_COMENZI` și sunt în categorii similare cu produsele care conțin în denumire cuvântul 'router'
- b) nu se actualizează niciun produs deoarece subcererea nu returnează nimic
- c) se actualizează toate produsele indiferent de categorie deoarece subcererea returnează întotdeauna aceeași valoare
- d) se afișează o eroare deoarece subcererea din IN poate returna mai multe categorii și operatorul IN nu este corect utilizat

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: a)**

**Explicație detaliată:**

Comanda UPDATE conține două condiții în WHERE:

**Condiția 1 (subcerere corelată):**
```sql
pret_lista > (SELECT AVG(pret) FROM rand_comenzi WHERE id_produs = p.id_produs)
```

- Returnează lista categoriilor produselor care au „router" în denumire.
- `IN` este perfect valid pentru subcereri care returnează **mai multe rânduri** (multi-row subquery).

Variantele greșite:
- b) Falsă — dacă nu există produse cu AVG, comparația cu NULL returnează NULL (condiția nu e îndeplinită), dar produse pot exista.
- c) Falsă — subcererea corelată returnează valori diferite pentru fiecare produs.
- d) Falsă — `IN` este proiectat exact pentru subcereri multi-row.

</details>

## Întrebarea 6

**Cerința:** Programul C++:

```cpp
#include <iostream>
using namespace std;

class A {
public:
    A() { cout << "A"; }
    ~A() { cout << "A"; }
};

class B : public A {
public:
    B() { cout << "B"; }
    ~B() { cout << "B"; }
};

class C : public B {
public:
    C() { cout << "C"; }
    ~C() { cout << "C"; }
};

int main() {
    C c;
    return 0;
}
```

**Variante:**
- a) ABCCBA
- b) ABCABC
- c) CBAABC
- d) CBACBA

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: a) ABCCBA**

**Explicație detaliată:**

Ierarhia de moștenire: `C` derivă din `B`, care derivă din `A`.

**La crearea obiectului `C c;` — ordinea constructorilor:**
Constructorii se apelează **de la bază spre derivat**:
1. `A()` → afișează `A`
2. `B()` → afișează `B`
3. `C()` → afișează `C`

Output parțial: `ABC`

**La distrugerea obiectului (ieșire din `main`) — ordinea destructorilor:**
Destructorii se apelează **invers față de constructori** (de la derivat spre bază):
1. `~C()` → afișează `C`
2. `~B()` → afișează `B`
3. `~A()` → afișează `A`

Output parțial: `CBA`

**Output total: `ABCCBA`**

Regula de reținut: **constructori de la bază la derivat, destructori invers**.

</details>

## Întrebarea 7

**Cerința:** Se consideră secvența SQL-Oracle:

```sql
CREATE TABLE angajati_noi AS SELECT * FROM angajati WHERE 1=2;

INSERT INTO angajati_noi
SELECT * FROM angajati WHERE data_angajare > TRUNC(sysdate, 'year');

ALTER TABLE angajati_noi ADD PRIMARY KEY (id_angajat);

ROLLBACK;
```

**Variante:**
- a) din cauza ROLLBACK, tabela nou creată nu va avea nicio înregistrare
- b) comanda INSERT va genera eroare din cauza sintaxei incorect utilizate
- c) tabela nou creată va conține înregistrările aferente angajaților din anul curent
- d) se va genera eroare din cauza tentativei de duplicare a restricției de cheie primară

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: c)**

**Explicație detaliată:**

Cheia înțelegerii este **comportamentul DDL în Oracle față de ROLLBACK**:

**Regula fundamentală:** Comenzile DDL (CREATE, ALTER, DROP) emit automat un **COMMIT implicit** înainte și după execuție. Aceasta înseamnă că orice tranzacție DML în curs este **comisă automat** când se execută un DDL.

**Trasarea execuției:**

1. `CREATE TABLE angajati_noi AS SELECT * FROM angajati WHERE 1=2;`
   - DDL → COMMIT implicit → tabela creată, goală (1=2 e mereu false).

2. `INSERT INTO angajati_noi SELECT * FROM angajati WHERE data_angajare > TRUNC(sysdate, 'year');`
   - DML → inserează angajații angajați în anul curent. Tranzacție deschisă.

3. `ALTER TABLE angajati_noi ADD PRIMARY KEY (id_angajat);`
   - **DDL → COMMIT implicit al INSERT-ului de la pasul 2!**
   - Adaugă constrângerea de cheie primară.

4. `ROLLBACK;`
   - Nu mai există nicio tranzacție de dat înapoi (totul a fost comis de DDL-uri).

**Concluzie:** Tabela `angajati_noi` va conține angajații din anul curent, cu cheie primară adăugată.

Variantele greșite:
- a) Falsă — ROLLBACK nu poate anula ce a fost comis de DDL.
- b) Falsă — INSERT cu subcerere este sintaxă validă.
- d) Falsă — `angajati_noi` are `id_angajat` ca PK potențial valid dacă valorile sunt unice.

</details>

## Întrebarea 8

**Cerința:** Fie tabelele `CLIENTI`, `RAND_COMENZI`, `COMENZI`. Se consideră comanda:

```sql
CREATE OR REPLACE VIEW v_comenzi AS
SELECT cl.nume_client, EXTRACT(MONTH FROM data) luna, c.id_comanda,
       ROUND(SUM(rc.pret * rc.cantitate), 2) valoare_comanda
FROM clienti cl, rand_comenzi rc, comenzi c
WHERE cl.id_client = c.id_client
  AND rc.id_comanda = c.id_comanda
  AND EXTRACT(MONTH FROM data) <= 07
GROUP BY cl.nume_client, EXTRACT(MONTH FROM data), c.id_comanda
HAVING SUM(rc.pret * rc.cantitate) >= 3000
ORDER BY 1, 2;
```

**Variante:**
- a) clauza ORDER BY este incorect utilizată
- b) crearea unei tabele virtuale prin care se vor selecta comenzile din luna iulie doar din anul curent
- c) crearea unei tabele virtuale prin care se vor selecta comenzile cu valoarea mai mare de 3000 până în luna iulie (inclusiv) indiferent de an
- d) se va afișa o eroare deoarece EXTRACT nu se poate utiliza în GROUP BY

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: c)**

**Explicație detaliată:**

Analiza comenzii pas cu pas:

**JOIN-ul (sintaxă implicită Oracle):**
- `clienti cl, rand_comenzi rc, comenzi c` cu condiții WHERE → echivalent cu INNER JOIN.

**Filtrarea `EXTRACT(MONTH FROM data) <= 07`:**
- Extrage **luna** din coloana `data` și verifică dacă este ≤ 7 (iulie).
- **Important:** Nu există filtrare pe AN — `EXTRACT(YEAR FROM data)` nu apare nicăieri.
- Prin urmare, sunt incluse comenzi din lunile 1-7 ale **oricărui an** din baza de date.

**GROUP BY și HAVING:**
- Grupare după client, lună, id_comandă.
- `HAVING SUM(...) >= 3000` → păstrează doar comenzile cu valoare ≥ 3000.

**ORDER BY 1, 2:**
- Referire la pozițiile coloanelor din SELECT (1 = `nume_client`, 2 = `luna`).
- Aceasta este validă în Oracle în cadrul unui VIEW.

Variantele greșite:
- a) Falsă — ORDER BY cu poziții numerice este valid în Oracle.
- b) Falsă — nu există filtrare pe an, deci nu e „doar din anul curent".
- d) Falsă — EXTRACT poate fi folosit în GROUP BY.

</details>

## Întrebarea 9

**Cerința:** Se consideră procedura PL/SQL:

```sql
CREATE OR REPLACE PROCEDURE sal_mediu(p_sal_mediu OUT NUMBER)
AS
BEGIN
  SELECT AVG(salariul) INTO p_sal_mediu FROM angajati WHERE 1=2;
END;
/
```

**Variante:**
- a) în cadrul comenzilor din corpul procedurii se utilizează o clauză de grupare
- b) se generează eroare deoarece nicio înregistrare nu îndeplinește condiția
- c) se generează eroare deoarece lipsește comanda RETURN
- d) după apelul procedurii, parametrul va primi valoarea NULL

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: d)**

**Explicație detaliată:**

**Ce face `WHERE 1=2`?**
- Condiția `1=2` este mereu falsă → **niciun rând nu este returnat** din tabelă.

**Comportamentul `AVG()` pe zero rânduri:**
- Funcțiile de agregare (`AVG`, `SUM`, `COUNT(*)`...) aplicat pe **zero rânduri** returnează **NULL** (excepție: `COUNT(*)` returnează 0).
- Prin urmare, `SELECT AVG(salariul) INTO p_sal_mediu FROM angajati WHERE 1=2` **nu aruncă excepția `NO_DATA_FOUND`** — returnează NULL.

**Comparație cu SELECT simplu:**
```sql
SELECT salariul INTO v FROM angajati WHERE 1=2; -- Aruncă NO_DATA_FOUND!
```
Dar cu `AVG()`:
```sql
SELECT AVG(salariul) INTO v FROM angajati WHERE 1=2; -- Returnează NULL, fără eroare
```

**Concluzie:** `p_sal_mediu` va primi valoarea NULL.

Variantele greșite:
- a) Parțial adevărat (AVG este funcție de grup), dar formularea este înșelătoare; nu este varianta „corectă" completă.
- b) Falsă — AVG pe zero rânduri returnează NULL, nu eroare.
- c) Falsă — `RETURN` nu este necesar în proceduri (este obligatoriu în funcții).

</details>

## Întrebarea 10

**Cerința:** Programul C:

```c
#include <stdio.h>
void main() {
    int a = 1, b = 3;
    while (1) {
        a *= --b;
        if (CONDITIE) break;
    }
    printf("%d", a);
}
```
Care este expresia cu care trebuie să se substituie `CONDITIE` astfel încât programul să afișeze `0`?

**Variante:**
- a) b > 0
- b) b == 0
- c) b != 0
- d) b >= 0

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: b) b == 0**

**Explicație detaliată:**

Trasăm execuția cu `CONDITIE = (b == 0)`:

| Iterație | `--b` | `b` după | `a *= b` → `a` | `b == 0`? |
|----------|-------|----------|----------------|-----------|
| 1        | 3→2   | 2        | 1 * 2 = 2      | 2 == 0? NU |
| 2        | 2→1   | 1        | 2 * 1 = 2      | 1 == 0? NU |
| 3        | 1→0   | 0        | 2 * 0 = **0**  | 0 == 0? **DA → break** |

Se afișează `a = 0`. ✓

**Verificarea celorlalte variante:**

- **a) b > 0:** La iterația 1, b=2 > 0 → break imediat, a=2. Afișează 2. ✗
- **c) b != 0:** La iterația 1, b=2 ≠ 0 → break imediat, a=2. Afișează 2. ✗
- **d) b >= 0:** La iterația 1, b=2 ≥ 0 → break imediat, a=2. Afișează 2. ✗

Singura variantă care permite ajungerea la `a=0` este când bucla continuă până când `b` devine 0, adică condiția `b == 0`.

</details>

## Întrebarea 11

**Cerința:** Programul C++:

```cpp
#include <iostream>
using namespace std;

class A {
public:
    A() { }
    A(int) { cout << "Copiere "; }
    A(const A&) { cout << "Copiere "; }
    A& operator=(A) { cout << "Atribuire "; return *this; }
    A& operator=(int) { cout << "Atribuire "; return *this; }
};

int main() {
    A ob1;
    A ob2 = ob1, ob3 = 7;
    ob3 = ob1;
    return 0;
}
```

**Variante:**
- a) Va afișa Copiere Atribuire Copiere Atribuire Copiere Atribuire
- b) Va afișa Copiere Copiere Atribuire
- c) Va afișa Copiere Copiere Copiere Atribuire
- d) Va genera erori de compilare

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: c) Va afișa Copiere Copiere Copiere Atribuire**

**Explicație detaliată:**

Analizăm fiecare instrucțiune:

**`A ob1;`**
- Apelează constructorul default `A()` → fără output.

**`A ob2 = ob1;`**
- Sintaxa `=` la inițializare (declarare) → **constructor de copiere** `A(const A&)`.
- Output: `Copiere`

**`A ob3 = 7;`**
- Sintaxa `=` la inițializare cu `int` → **constructor cu int** `A(int)`.
- Output: `Copiere`

**`ob3 = ob1;`**
- `ob3` există deja → aceasta este **atribuire**, nu inițializare → `operator=(A)`.
- Parametrul `operator=(A)` se primește **prin valoare** → constructorul de copiere este apelat mai întâi pentru a copia `ob1` în parametru!
- Deci: **constructor de copiere** `A(const A&)` → `Copiere`, apoi **operator=** → `Atribuire`.

**Output total:** `Copiere Copiere Copiere Atribuire`

**Distincția cheie:**
- `A ob = altceva;` → **inițializare** = constructor (nu operator=)
- `ob_existent = altceva;` → **atribuire** = operator= (dar parametrul by-value invocă copy constructor)

</details>

## Întrebarea 12

**Cerința:** Care dintre următoarele afirmații este corectă?

**Variante:**
- a) în diagrama entitate-asociere, un atribut compus are valoarea calculată pe baza valorilor altor atribute
- b) în diagrama entitate-asociere, asocierea recursivă se realizează între o entitate și ea însăși
- c) în modelul relațional, restricția entității se implementează prin UNIQUE
- d) conceptul de tuplu din teoria relațională are corespondent conceptul de coloană la implementarea unei tabele

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: b)**

**Explicație detaliată:**

- **b) ADEVĂRAT:** O **asociere recursivă** (sau auto-referențială) conectează o entitate cu ea însăși. Exemplu clasic: entitatea `ANGAJAT` are o asociere „conduce" cu `ANGAJAT` (un angajat este condus de alt angajat).

**Variantele greșite:**

- **a) FALSĂ:** Un **atribut compus** este format din mai multe sub-atribute (ex: Adresa = Stradă + Nr + Oraș). Valoarea calculată pe baza altor atribute se numește **atribut derivat** (marcat cu linie punctată în ERD).

- **c) FALSĂ:** Restricția de entitate (entity integrity) se referă la faptul că **cheia primară nu poate fi NULL** — se implementează prin `PRIMARY KEY`. `UNIQUE` implementează **cheia alternativă** (alternate key / candidate key).

- **d) FALSĂ:** Un **tuplu** din teoria relațională corespunde unui **rând (înregistrare)** dintr-o tabelă, nu unei coloane. O **coloană** corespunde unui **atribut**.

| Concept relațional | Implementare SQL |
|--------------------|-----------------|
| Relație            | Tabelă          |
| Tuplu              | Rând            |
| Atribut            | Coloană         |

</details>

```sql
ROUND(45.923, 0) → 46
ROUND(45.923, -1) → 50
```

**2. ROUND pentru date:**
```sql
ROUND(SYSDATE, 'YEAR')  → 01-JAN-YYYY (rotunjit la cel mai apropiat an)
ROUND(SYSDATE, 'MONTH') → prima zi a lunii celei mai apropiate
ROUND(SYSDATE, 'DAY')   → prima zi a săptămânii celei mai apropiate
```

**Variantele greșite:**
- a) Falsă — funcționează și pe DATE.
- c) Falsă — `ROUND(DATE, 'MONTH')` nu returnează luna curentă ca număr; returnează o dată.
- d) Falsă — `ROUND(sysdate, 'day')` returnează **ziua de duminică a săptămânii curente** (nu prima zi a anului). Prima zi a anului se obține cu `TRUNC(sysdate, 'YEAR')`.

</details>

## Întrebarea 27

**Cerința:** Care dintre următoarele afirmații este corectă?

**Variante:**
- a) parametrii unei proceduri PL/SQL trebuie să fie întotdeauna de tip scalar
- b) o funcție PL/SQL trebuie să conțină cel puțin o instrucțiune RETURN
- c) o funcție PL/SQL poate fi întotdeauna apelată într-o expresie SQL
- d) o funcție PL/SQL nu poate primi decât parametri de intrare (tipul IN)

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: b)**

**Explicație detaliată:**

- **b) ADEVĂRAT:** O funcție PL/SQL **obligatoriu** trebuie să conțină cel puțin un `RETURN` care returnează valoarea de tipul declarat. Fără `RETURN`, compilatorul generează eroare. (Spre deosebire de proceduri, unde `RETURN` este opțional și nu returnează valori.)

**Variantele greșite:**

- **a) FALSĂ:** Parametrii pot fi de tip scalar (NUMBER, VARCHAR2, DATE), dar pot fi și de tip **compus** (RECORD, TABLE, colecții Oracle).

- **c) FALSĂ:** O funcție PL/SQL **nu poate** fi apelată în SQL dacă:
  - Conține DML (INSERT/UPDATE/DELETE) fără PRAGMA AUTONOMOUS_TRANSACTION
  - Conține comenzi de tranzacție (COMMIT, ROLLBACK)
  - Are parametri OUT sau IN OUT

- **d) FALSĂ:** Funcțiile PL/SQL pot avea parametri de tip `IN`, `OUT` și `IN OUT`, deși buna practică recomandă doar `IN` pentru funcții (valorile de ieșire se returnează prin RETURN).

</details>

## Întrebarea 28

**Cerința:** Programul C:

```c
struct s { int x[2]; };
union u { int x; int y; };

void main() {
    struct s a;
    union u b;
}
```

Care afirmație este adevărată despre dimensiunile variabilelor `a` și `b`?

**Variante:**
- a) Mărimea lui `b` este de trei ori mai mare decât `a`
- b) Mărimea lui `b` este dublul mărimii lui `a`
- c) Mărimea lui `a` este egală cu mărimea lui `b`
- d) Mărimea lui `a` este dublul mărimii lui `b`

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: d)**

**Explicație detaliată:**

**Struct `s`:**
```c
struct s { int x[2]; };
```
- Conține un array de 2 int-uri.
- `sizeof(struct s) = 2 × sizeof(int) = 2 × 4 = **8 bytes**`

**Union `u`:**
```c
union u { int x; int y; };
```
- Un union alocă spațiu egal cu **cel mai mare câmp** (toate câmpurile partajează același spațiu de memorie).
- `sizeof(union u) = max(sizeof(int), sizeof(int)) = sizeof(int) = **4 bytes**`

**Comparație:**
- `sizeof(a) = 8 bytes`
- `sizeof(b) = 4 bytes`
- `sizeof(a) = 2 × sizeof(b)`

**Concluzie:** Variabila `a` are dimensiunea dublă față de `b`.

**Principiul union:** Toate câmpurile unui union ocupă **același spațiu de memorie** — scrierea unui câmp suprascrie ceilalți. Este util pentru conversii de tip sau economie de memorie.

</details>

## Întrebarea 29

**Cerința:** În C++, o clasă poate conține toate secțiunile (private, protected și public) în același timp?

**Variante:**
- a) da
- b) da, numai dacă este clasă abstractă
- c) da, numai dacă este o clasă de bază într-o relație de derivare
- d) nu, acest lucru nu este permis

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: a) da**

**Explicație detaliată:**

În C++, o clasă poate conține oricâte secțiuni de acces, în orice ordine și de câte ori dorim:

```cpp
class Exemplu {
private:
    int x;           // secțiune private

protected:
    int y;           // secțiune protected

public:
    Exemplu() {}     // secțiune public

private:             // a doua secțiune private — valid!
    int z;

public:
    void metoda();   // a doua secțiune public — valid!
};
```

Nu există nicio restricție care să lege prezența tuturor celor trei secțiuni de:
- tipul clasei (abstractă sau concretă)
- rolul clasei (bază sau derivată)

Variantele b), c), d) sunt toate false — constrângerile respective nu există în C++.

</details>

## Întrebarea 30

**Cerința:** Programul C:

```c
#include <stdio.h>
void main() {
    int a[][2] = { {1,2}, {1,0} };
    int v[] = { 10, 20, 30 };
    printf("\n%d", v[a[1][1]] + v[**a+1]);
}
```

**Variante:**
- a) Afișează 30
- b) Afișează 40
- c) Afișează 10
- d) Generează o eroare datorită expresiei incorecte

<details>
<summary>🔍 Răspuns și explicație</summary>

**Răspuns corect: b) Afișează 40**

**Explicație detaliată:**

**Valorile matricei `a`:**
```
a[0][0]=1  a[0][1]=2
a[1][0]=1  a[1][1]=0
```

**Vectorul `v`:**
```
v[0]=10  v[1]=20  v[2]=30
```

**Evaluarea expresiei `v[a[1][1]] + v[**a+1]`:**

**Partea 1: `v[a[1][1]]`**
- `a[1][1] = 0`
- `v[0] = 10`

**Partea 2: `v[**a+1]`**
- `**a` = dereferențierea dublă a matricei `a`
- `a` → pointer la `a[0]` (primul rând)
- `*a` → pointer la `a[0][0]` = pointer la primul element al primului rând
- `**a` → valoarea de la `a[0][0] = 1`
- `**a + 1` = `1 + 1 = 2`
- `v[2] = 30`

**Suma:** `v[0] + v[2] = 10 + 30 = **40**`

**Notă tehnică:** `**a` funcționează deoarece `a` este un array 2D; în C, `a` decay-uiește la pointer la primul rând, `*a` decay-uiește la pointer la primul element, `**a` dă valoarea elementului `a[0][0]`.

</details>
